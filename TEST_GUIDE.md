# GlowUp Test Guide

A reference for advanced developers (and future Claude instances) explaining
what the higher-level tests prove, what failure modes they guard against, and
how the test infrastructure works.

## Test Architecture Overview

Tests are organized in three tiers:

| Tier | Files | What it tests | Speed |
|------|-------|---------------|-------|
| **Unit** | `test_audit_regressions.py`, `test_routing.py`, `test_config.py`, `test_virtual_multizone.py` | Individual functions, source-level invariants, symbol correctness | < 1s |
| **Use-case** | `test_use_cases.py` | Full Controller → Engine → Emitter pipeline with real effects | ~19s |
| **Distributed** | `test_distributed.py` | Multi-machine SOE pipeline, wire protocols, MIDI/audio with known fixtures | < 1s |

### Running tests

```bash
# All tests (fast tiers):
python3 -m unittest test_audit_regressions test_routing test_config \
    test_virtual_multizone test_distributed

# Use-case tests (slower — runs real effect engines):
python3 -m unittest test_use_cases -v

# Everything:
python3 -m unittest discover -p "test_*.py" -v
```

## Test Infrastructure

### RecordingEmitter (test_use_cases.py)

The key to hardware-free pipeline testing. Implements the full `Emitter`
interface but writes frames to a list instead of UDP sockets.

**What it captures:**
- `zone_frames` — every multizone frame (list of HSBK tuples)
- `color_frames` — every single-zone color
- `power_calls` — every power_on/power_off with duration

**Why it matters:** The Engine's send thread writes to emitters from a
background thread. `RecordingEmitter` is thread-safe (internal lock) so
tests can read captured frames while the engine is still running.

**If a test using RecordingEmitter fails:** The problem is in the pipeline
itself (Engine, Controller, Effect, or the emitter dispatch logic), not in
network transport or hardware. This is the whole point — isolating the
pipeline from the physical world.

### Test Fixtures (tests/fixtures/)

Deterministic binary files generated by `generate_fixtures.py`. Regenerable
at any time with identical content — no external downloads, no git-lfs.

| Fixture | Format | Content | Purpose |
|---------|--------|---------|---------|
| `test_scale.mid` | SMF Type 0 | C major scale (C4-C5), 8 notes, vel=100, 120 BPM | MIDI parser validation |
| `test_440hz.wav` | PCM 16-bit mono | 440 Hz sine, 1 second, 44100 Hz | Audio extractor, RMS validation |
| `test_c_major_chord.wav` | PCM 16-bit mono | C4+E4+G4 simultaneous, 1 second | FFT frequency detection |
| `test_440hz.mp3` | MP3 128kbps | 440 Hz sine (from WAV via ffmpeg) | MP3 codec pipeline |
| `test_corrupt.mp3` | Random bytes | 4096 bytes of deterministic garbage | Codec error handling |

**If a fixture is missing:** `test_distributed.py` auto-regenerates all
fixtures on import. The MP3 fixture requires ffmpeg at generation time —
if ffmpeg is absent, MP3 tests are skipped (not failed).

## Use-Case Tests (test_use_cases.py)

These tests prove that the effect engine works end-to-end as a user would
experience it. Each test category maps to a real usage scenario.

### UC-1: Basic Playback

**What it proves:** Every registered effect can start, produce at least one
frame of valid HSBK data, and stop without crashing.

**`test_every_effect_produces_frames`**
- Iterates the entire effect registry
- Plays each effect for 200ms on a 36-zone RecordingEmitter
- Validates HSBK component ranges on the first frame
- Effects that need special setup (media bus, external signals) are skipped
  and reported to stderr — not failures

**Failure means:**
- An effect's `render()` is crashing or returning malformed data
- The Engine's render thread is dying silently
- An effect registered with a broken name or missing class

**`test_single_zone_emitter`**
- Validates that a 1-zone, non-multizone emitter receives `send_color()`
  calls (not `send_zones()`)

**Failure means:** The Engine's send dispatch is broken for the single-zone
code path. This would affect individual bulbs (A19, BR30) as opposed to
strips and beams.

### UC-2: Lifecycle

**What these prove:** The effect engine handles state transitions correctly —
the kinds of operations a phone app or scheduler performs repeatedly.

**`test_stop_halts_frame_production`**
- Plays an effect, stops it, waits, checks frame count hasn't increased

**Failure means:** The engine's stop mechanism is broken — threads are still
running after `stop()`. This would cause resource leaks and ghost animations
on real hardware.

**`test_hot_swap_effect`**
- Plays "breathe", then plays "cylon" without stopping first
- Verifies frames were produced and the status reports "cylon"

**Failure means:** Hot-swapping effects (what happens when a user changes
effects from the phone app) crashes or deadlocks. This is a critical
real-world scenario — the scheduler hot-swaps effects on time boundaries.

**`test_double_stop_is_safe`**
- Calls `stop()` twice in succession

**Failure means:** The engine doesn't handle idempotent stop, which would
crash the scheduler when it tries to stop an already-stopped device.

**`test_play_after_stop`**
- Stop, then play again — verifies the engine can be reused

**Failure means:** The engine's threads are not properly cleaned up on stop,
preventing restart. Would require server restart to play effects again.

### UC-3: Signal Binding (Media-Reactive)

**What these prove:** External signals (audio FFT, sensors, MIDI) can drive
effect parameters in real time without the effect knowing about the binding.

**`test_signal_drives_effect_param`**
- Binds "speed" to a test signal on the SignalBus
- Writes values to the bus before and during playback

**Failure means:** The binding resolver in `Engine._resolve_bindings()` is
broken. Audio-reactive effects would stop responding to music. This is the
core of the media pipeline.

**`test_missing_signal_uses_default`**
- Binds to a signal name that doesn't exist on the bus

**Failure means:** The bus returns a crash instead of the default value.
Would affect any effect started before its signal source is online (common
in distributed setups where nodes start at different times).

**`test_array_signal_reduced`**
- Writes an 8-element FFT band array with `reduce='mean'`

**Failure means:** The array reduction logic (max/mean/sum) is broken.
Audio-reactive effects receive arrays from the FFT extractor — if reduction
crashes, all audio-reactive effects fail.

### UC-4: Multi-Device Fan-Out

**What these prove:** Virtual groups (multiple physical devices addressed as
one) distribute frames correctly.

**`test_virtual_group_distributes_zones`**
- Creates 3 emitters with 12 zones each, wrapped in a VirtualMultizoneEmitter
- Plays an effect on the 36-zone virtual canvas
- Verifies each member received frames with its own zone count (12)

**Failure means:** Zone routing in `VirtualMultizoneEmitter.send_zones()` is
broken. Zones would go to the wrong bulbs — e.g., the left third of an
animation appearing on the right strip. This is visually obvious but hard to
debug without tests.

**`test_mixed_single_and_multizone`**
- Virtual group with a 30-zone strip + a 1-zone bulb
- Verifies the strip gets `send_zones()` and the bulb gets `send_color()`

**Failure means:** The fan-out logic doesn't handle mixed emitter types.
Groups containing both string lights and individual bulbs would break.

### UC-5: Frame Correctness

**`test_all_effects_produce_legal_hsbk`**
- Spot-checks first 5 frames from each effect
- Validates all HSBK components: hue ∈ [0, 65535], sat ∈ [0, 65535],
  brightness ∈ [0, 65535], kelvin ∈ [1500, 9000]

**Failure means:** An effect is producing out-of-range values. LIFX firmware
silently clamps, but this can cause unexpected colors (e.g., kelvin=0 would
be interpreted differently across firmware versions).

**`test_zones_per_bulb_replication`**
- Plays with `zones_per_bulb=3` (string light mode)
- Verifies each color is replicated exactly 3 times consecutively

**Failure means:** The zpb replication logic in `Engine._render_thread()` is
broken. String lights (which have 3 internal zones per visible bulb) would
show non-uniform colors within a single bulb — a visible artifact.

### UC-6: Frame Callback

**`test_callback_receives_frames`**
- Registers a callback (the simulator uses this exact mechanism)
- Verifies it receives frames

**Failure means:** The simulator preview would show nothing.

**`test_callback_exception_does_not_crash_engine`**
- Registers a callback that raises RuntimeError every time
- Verifies the engine keeps producing frames for the emitter

**Failure means:** A buggy simulator or monitoring callback would kill the
entire render pipeline. This was an actual risk before we added exception
logging to the callback path (the bare-except fix).

### UC-7: Runtime Parameter Update

**`test_update_params_while_running`**
- Changes `speed` from 2.0 to 8.0 while the effect is playing

**Failure means:** The phone app's parameter sliders would crash the engine
when the user adjusts them during playback.

**`test_update_unknown_param_is_safe`**
- Passes a param name that the effect doesn't have

**Failure means:** The API would crash when receiving unexpected parameters.
Since the phone app may be a different version than the server, unknown
params must be silently ignored.

## Distributed Tests (test_distributed.py)

These tests prove the multi-machine agent system works — capability
registration, work assignment, wire protocols, and audio/MIDI processing
with known data.

### 1. NodeCapability & NodeHealth

**What they prove:** Nodes can describe themselves and their health in a
format that survives MQTT serialization.

**Failure means:**
- `round_trip_json` — A field is lost during serialization. The orchestrator
  would see incomplete capabilities and misroute work.
- `from_json_malformed` — A corrupt MQTT message crashes the orchestrator
  instead of being silently dropped. In production, broker restarts and
  network glitches produce garbage payloads.
- `timestamp_auto_generated` — Nodes without explicit timestamps would be
  treated as "time unknown," breaking staleness detection.

### 2. WorkAssignment & SignalBinding

**What they prove:** Work instructions survive the orchestrator → MQTT →
worker serialization path.

**Failure means:**
- `compute_assignment_round_trip` — The worker receives a garbled assignment.
  An AudioExtractor starts with wrong bands count or wrong input port.
- `emitter_assignment_round_trip` — A remote emitter (e.g., audio output on
  a Mac) receives wrong config. The theremin synth plays at wrong volume.
- `is_emitter_assignment` — The worker can't distinguish compute from emitter
  work. It would try to instantiate an operator when it should create an
  emitter, or vice versa.

### 3. Orchestrator Fleet Management

**What they prove:** The orchestrator correctly tracks node state, selects
appropriate nodes for work, and manages the UDP port pool.

**`test_node_registration`** — A new node appears in the fleet after
publishing its capability.

**`test_node_goes_offline`** — The orchestrator marks nodes offline when
they disconnect (LWT message). Failure means work gets assigned to dead
nodes.

**`test_select_compute_node_prefers_gpu`** — GPU nodes are preferred for
compute work (FFT, ML inference). Failure means the Pi gets assigned GPU
work it can't handle, while the ML box sits idle.

**`test_port_allocation`** — UDP ports are allocated uniquely from a bounded
pool and can be recycled. Failure means port conflicts when two operators
share a machine, causing data crosstalk.

**`test_assign_work_publishes_mqtt`** — The assignment actually gets
published. Failure means the orchestrator silently drops assignments.

### 4. UDP Wire Protocol

**What they prove:** Binary signal frames (audio PCM, FFT bands) survive
network serialization without corruption.

**`test_signal_data_round_trip`** — 8 float32 FFT bands pack and unpack
correctly. Failure means garbled audio data — effects react to noise.

**`test_pcm_audio_round_trip`** — 4410 int16 PCM samples (100ms of audio).
Failure means audio chunks arrive corrupted at the extractor.

**`test_malformed_data_returns_none`** — Short, truncated, and wrong-magic
packets return None. Failure means the receiver loop crashes on a single
bad packet, killing the entire data pipeline.

**`test_sequence_wraps_at_32bit`** — Sequence numbers wrap correctly.
Failure means packet ordering breaks after ~49 days of continuous operation
at 20 Hz.

**`test_oversized_frame_raises`** — Frames exceeding UDP's 65507-byte limit
are rejected at pack time. Failure means silent truncation — half an audio
chunk arrives, producing clicks and pops.

### 5. MIDI Parser (Known Fixture)

**What they prove:** The MIDI parser correctly decodes Standard MIDI Files
against a fixture with known, verifiable content.

**`test_note_values_are_c_major_scale`** — The 8 note-on events are exactly
C4, D4, E4, F4, G4, A4, B4, C5 (MIDI 60, 62, 64, 65, 67, 69, 71, 72).
Failure means the parser is miscalculating note numbers — the MIDI light
bridge would map wrong colors.

**`test_velocity_is_uniform`** — All velocities are 100. Failure means
velocity parsing is broken — brightness would be wrong in MIDI-driven
light effects.

**`test_events_are_time_ordered`** — Events are in ascending time_s order.
Failure means the tempo-to-wallclock conversion is broken — notes play
out of order.

**`test_event_to_dict_round_trip`** — Events serialize to JSON. Failure
means the MQTT sensor can't publish events — the entire MIDI-over-MQTT
pipeline breaks.

### 6. Audio Extraction (Known Fixture)

**`test_pcm_rms_matches_440hz_sine`** — The 440 Hz fixture has a known
RMS of ~23170 (0.707 × 32767). Failure means the WAV was generated
incorrectly or the test math is wrong — baseline invalidated.

**`test_audio_extractor_produces_signals`** — Feed the 440 Hz sine through
the real AudioExtractor; verify it writes nonzero RMS and band energy to
the SignalBus. Failure means the extractor's FFT pipeline is broken — no
audio-reactive effects would work anywhere in the fleet.

### 6b. MP3 Codec Handling

**What they prove:** The system handles MP3 input correctly and degrades
gracefully when codecs fail.

**`test_ffmpeg_mp3_decode_to_pcm`** — ffmpeg decodes the MP3 fixture to
raw PCM. Failure means the audio sensor's ffmpeg pipeline is broken —
no MP3 sources would work (most users will have MP3 music).

**`test_ffmpeg_rejects_corrupt_mp3`** — ffmpeg returns nonzero for garbage.
Failure means corrupt files produce silent garbage data instead of visible
errors — effects would react to noise that looks like real audio.

**`test_corrupt_mp3_does_not_crash_extractor`** — Feeding garbage bytes
to AudioExtractor doesn't crash. The extractor processes raw PCM; garbage
bytes produce garbage signals but no segfault or unhandled exception.
Failure means a single corrupt audio packet kills the entire audio pipeline
on a compute node.

### 6c. FFT Chord Detection

**What they prove:** The FFT correctly resolves individual frequencies
within a complex waveform.

**`test_fft_detects_chord_frequencies`** — The C major chord fixture has
energy peaks at 261.6, 329.6, and 392.0 Hz, and minimal energy at 500 Hz.
This is a ground-truth FFT test: we know exactly what frequencies are
present because we generated the waveform mathematically. Failure means
the FFT window, bin mapping, or magnitude calculation is wrong — band
assignments in AudioExtractor would be incorrect.

**`test_extractor_sees_multiple_bands`** — The chord produces energy in
more than one frequency band. Failure means the band binning in
AudioExtractor is collapsing all energy into a single band — multi-band
audio-reactive effects (bass-hue, treble-brightness) would all respond
to the same signal.

### 7. SignalBus Core

**What they prove:** The central data exchange layer works correctly.

These are simple read/write tests, but the bus is the *single point of
failure* for the entire media pipeline. Every signal — audio, MIDI, sensor,
operator output — flows through it.

**`test_read_unknown_returns_default`** — Missing signals return 0.0.
Failure means effects crash on startup before signal sources are connected.

**`test_overwrite_updates_value`** — Writes are visible to subsequent reads.
Failure means signal values are stale — effects show old audio data.

### 8. Assignment-to-Signal Flow

**What they prove:** The data structures that wire the distributed pipeline
together are consistent end-to-end.

These tests validate the *contract* between orchestrator and worker: the
assignment JSON describes inputs and outputs, the bus carries the data, and
the signal names match. They don't test actual MQTT transport (that needs
a broker), but they prove the plumbing is correctly shaped.

## Failure Triage Quick Reference

| Symptom | Start here |
|---------|-----------|
| Effect produces no frames | UC-1, check effect's `render()` |
| Effect crashes on start | UC-1, check `create_effect()` and params |
| Frames still arriving after stop | UC-2 `test_stop_halts_frame_production` |
| Hot-swap deadlocks | UC-2 `test_hot_swap_effect`, check Engine._lock |
| Audio-reactive effects unresponsive | UC-3, check SignalBus and bindings |
| Wrong zones on wrong strips | UC-4, check VirtualMultizoneEmitter routing |
| Out-of-range colors | UC-5, check effect's HSBK math |
| String lights show non-uniform bulbs | UC-5 `test_zones_per_bulb_replication` |
| Simulator shows nothing | UC-6 `test_callback_receives_frames` |
| Phone slider crashes server | UC-7 `test_update_params_while_running` |
| Node not appearing in fleet | Distributed 3, check capability JSON |
| Work assigned to wrong node | Distributed 3, check `select_compute_node` |
| Audio garbled across network | Distributed 4, check UDP protocol |
| MIDI notes out of order | Distributed 5, check time ordering |
| MP3 source silent | Distributed 6b, check ffmpeg pipeline |
| FFT bands all zero | Distributed 6/6c, check AudioExtractor |
| Signals not flowing between nodes | Distributed 7/8, check SignalBus |
